# StockAPI — Заказы (`/orders`)

HTTP-сервис **StockAPI** предназначен для интеграции внешних систем с исходящими заказами покупателей в 1С.

В учёте заказ представлен документом **Инвойс продажи** (`ИнвойсПродажи`, проформа «Заказ покупателя»). API создаёт, изменяет и возвращает такие проформы, резервирует остатки, формирует расходные накладные и отдаёт их печатные формы.

Статусы заказа хранятся в регистре **СостоянияИнвойсовПродажи** и возвращаются в полях `status`, `statusSetBy`, `statusDate`.

---

## Авторизация

Во всех запросах передавайте заголовок:

```
X-Api-Key: <токен учётной записи интеграции>
```

Токен соответствует реквизиту `ТокенАвторизации` справочника `УчетныеЗаписиИнтеграции`. Неактивная или неизвестная учётная запись → **401**.

Доступ ограничен настройками учётной записи:
- **склады** — табличная часть «Склады»;
- **контрагенты** — группа контрагентов учётной записи;
- **заказы** — видны только документы с реквизитом `УчетнаяЗапись`, совпадающим с ключом API.

Тело запросов и ответов — **JSON**, кодировка **UTF-8**.

Формат дат в ответах: `yyyy-MM-ddTHH:mm:ss`. В параметрах запроса даты передаются как `yyyy-MM-dd`.

## Эндпоинты

| Метод | URL | Назначение |
|-------|-----|------------|
| POST | `/orders` | Создание проформы с резервированием остатков |
| GET | `/orders` | Получение одного заказа или списка с пагинацией |
| PATCH | `/orders` | Полная замена заголовка и позиций |
| PUT | `/orders` | Добавление позиций |
| DELETE | `/orders` | Удаление позиций; при указании всех позиций — пометка заказа на удаление |
| GET | `/series` | Серии остатков по серийным позициям заказа |
| POST | `/delivery` | Создание и проведение расходной накладной по сериям |
| PATCH | `/delivery` | Блокировка отгрузки (статус «Собирается (заблокирован)»); требует расходную накладную |
| POST | `/status` | Установка признаков заказа: `blocked` / `canceled` / `readyToShip` |
| GET | `/delivery-notes` | PDF/Excel расходной накладной |

Вспомогательные эндпоинты: `GET /warehouses`, `GET /employees`, `GET /counterparties`, `GET /balances`.

## Идентификация заказа

| Поле JSON / query | Синонимы |
|-------------------|----------|
| `incomingNumber` | `incomingNo`, `incoming` |
| `orderNumber` | `orderNo`, `order` |

- **incomingNumber** — входящий номер клиента (реквизит `НомерВходящий` в 1С). Обязателен при **POST /orders**; должен быть уникален.
- **orderNumber** — номер документа в 1С (присваивается автоматически при создании).

Для поиска существующего заказа достаточно одного из номеров. Заказ должен принадлежать той же учётной записи API, что и ключ авторизации.

Дополнительно при поиске одного заказа можно указать `orderDate` или `date` — дату документа (`yyyy-MM-dd`).

## Пагинация (GET /orders)

При выборке списка заказов:

| Параметр | Синоним | Описание |
|----------|---------|----------|
| `page` | — | Номер страницы, с 1 (по умолчанию 1) |
| `pageSize` | `limit` | Размер страницы, 1–500 (по умолчанию 100) |

Ответ содержит: `page`, `pageSize`, `totalItems`, `totalPages`, `items`.

## Сотрудник (оператор склада)

Для **PATCH /orders**, **PUT /orders**, **DELETE /orders**, **PATCH /delivery**, **POST /status** можно передать исполнителя:

| Поле | Синоним | Описание |
|------|---------|----------|
| `employeeCode` | — | Числовой код сотрудника |
| `employeePin` | — | PIN сотрудника |

Параметры допускаются в **теле JSON** или в **query string** URL.

Сотрудник должен быть **оператором склада**: присутствовать в табличной части «Склады» справочника `Сотрудники` и работать на складе, разрешённом учётной записи API. Список доступных сотрудников — `GET /employees?warehouseCode=...`.

Если сотрудник указан, но не найден или не имеет доступа — запрос завершится ошибкой **400**.

## Статусы заказа

| Поле ответа | Описание |
|-------------|----------|
| `status` | Текущее состояние (основной статус документа) |
| `statusSetBy` | Кем установлен (сотрудник или пользователь 1С) |
| `statusDate` | Дата и время регистрации статуса |
| `blocked` | Признак «заказ заблокирован для сборки» |
| `canceled` | Признак «заказ отменён заказчиком» |
| `readyToShip` | Признак «заказ готов к отгрузке» |

Помимо основного статуса `status`, у заказа есть три **независимых булевых признака** — `blocked`, `canceled`, `readyToShip`. Каждый хранится со своей отметкой времени и может быть установлен независимо от остальных и от `status` через **POST /status** (см. ниже). Признак `readyToShip` дополнительно можно задать при **POST /orders** и **PATCH /orders**.

### Значения статуса (`status`)

| status (RU) | EN | Когда устанавливается |
|-------------|----|-----------------------|
| Зарегистрирован | Registered | Первое проведение проформы (**POST /orders**) |
| Изменен | Changed | Автоматически при **проведении** изменённого заказа — интерактивно в 1С (форма помечена как модифицированная) либо через **PATCH /orders** / **PUT /orders** |
| Собирается (заблокирован) | Picking up (blocked) | **PATCH /delivery** (требует существующую расходную накладную) |
| Готов к отгрузке | Ready for shipment | Вручную в интерфейсе 1С |
| Отгружен | Dismissed | Проведение расходной накладной с флагом «Окончательно» (**POST /delivery** или вручную в 1С) |
| Отменен | Canceled | **DELETE /orders** (все позиции) или отмена в 1С |

> Не путайте статус «Готов к отгрузке» (значение `status`, задаётся вручную в 1С) с признаком `readyToShip` (независимый флаг, задаётся через API).

### Ограничения редактирования по статусу

- **PATCH / PUT / DELETE /orders** запрещены, если по заказу уже создана **расходная накладная** (даже не проведённая) → **409**.
- **PATCH /delivery** недоступен, если заказ уже в статусе «Собирается», «Отгружен» или «Отменен», либо расходная накладная проведена → **409**.
- **POST /status** — если все переданные признаки (`blocked`/`canceled`/`readyToShip`) уже имеют запрошенные значения → **409**.

## Жизненный цикл заказа

```
POST /orders  →  Зарегистрирован (резерв остатков)
       ↓
PATCH|PUT|DELETE /orders  →  изменение состава → статус «Изменен» при проведении
       ↓
POST /status (blocked=true)  →  признак blocked = true (статус не меняется)
       ↓
POST /delivery  →  Расходная накладная (+ «Отгружен» при «Окончательно»)
       ↓
GET /delivery-notes  →  PDF / Excel накладной
```

Признаки `blocked`, `canceled`, `readyToShip` устанавливаются и снимаются через **POST /status** независимо друг от друга и от статуса `status` (передавайте `true`/`false`). `readyToShip` также можно задать сразу при **POST /orders** или **PATCH /orders**.

Удаление всех позиций через **DELETE /orders** помечает заказ на удаление (снятие резерва выполняется платформой 1С).

## POST /orders — создание заказа

**Тело запроса (основные поля):**

| Поле | Обяз. | Описание |
|------|-------|----------|
| `incomingNumber` | да | Входящий номер заказа клиента |
| `customer` / `customerCode` | да* | Покупатель: наименование или код контрагента |
| `warehouse` / `warehouseCode` / `warehouseId` | да | Склад из списка, доступного учётной записи |
| `items` | да | Массив позиций (см. ниже) |
| `reserve` / `reserved` | нет | Резервировать товар (по умолчанию **true**) |
| `employeeCode` / `employeePin` | нет | Оператор склада |
| `readyToShip` | нет | Признак готовности к отгрузке (`true`/`false`, `1`/`0`, `yes`/`no`) |

\* Контрагент должен входить в группу, разрешённую учётной записи API.

**Проверки при создании:**
- уникальность `incomingNumber`;
- наличие вида цен у контрагента (валюта и цены берутся из него);
- достаточность остатков на складе (остаток минус резерв);
- совпадение `price` каждой позиции с `salePrice` из **GET /balances**.

**Успешный ответ (200):**

```json
{
  "success": true,
  "incomingNumber": "PO-2026-001",
  "orderNumber": "MT-20260708-00042",
  "orderDate": "2026-07-08T10:15:00",
  "status": "Зарегистрирован",
  "statusSetBy": "Иванов И.И.",
  "statusDate": "2026-07-08T10:15:01",
  "reserved": true,
  "shipmentDate": "",
  "readyToShip": false
}
```

При дубликате `incomingNumber` — **400** с полями существующего заказа (`orderNumber`, `status`, `readyToShip`, …).

## GET /orders — получение заказа(ов)

**Режим одного заказа** — указан `orderNumber` и/или `incomingNumber`:

```
GET /orders?incomingNumber=PO-2026-001
GET /orders?orderNumber=MT-20260708-00042
GET /orders?incomingNumber=PO-2026-001&orderDate=2026-07-08
```

**Режим списка** — номер не указан, фильтр по контрагенту и периоду:

```
GET /orders?customerCode=1001&dateFrom=2026-07-01&dateTo=2026-07-31&page=1&pageSize=50
```

| Параметр | Синонимы | Описание |
|----------|----------|----------|
| `customer` | `counterparty`, `name` | Наименование покупателя |
| `customerCode` | `counterpartyCode`, `code` | Код контрагента |
| `dateFrom` | `from` | Начало периода по дате документа (`yyyy-MM-dd`) |
| `dateTo` | `to` | Конец периода по дате документа (`yyyy-MM-dd`) |
| `shipmentDateFrom` | — | Отбор по `ДатаОтгрузки`: с указанной даты (`yyyy-MM-dd`) |
| `shipmentDateTo` | — | Отбор по `ДатаОтгрузки`: по указанную дату (`yyyy-MM-dd`) |

Если контрагент не указан — возвращаются заказы всех контрагентов группы учётной записи. Период `dateFrom`/`dateTo` задаётся парой; при указании одной даты — **400**.

`shipmentDateFrom`/`shipmentDateTo` можно указывать по отдельности или вместе (в отличие от `dateFrom`/`dateTo`); при совместном указании `shipmentDateFrom` не может быть позже `shipmentDateTo` — иначе **400**.

**Структура элемента `items[]` в ответе:**

| Поле | Описание |
|------|----------|
| `orderNumber` | Номер в 1С |
| `incomingNumber` | Входящий номер |
| `orderDate` | Дата документа |
| `customer`, `customerCode` | Покупатель |
| `warehouse`, `warehouseCode` | Склад |
| `currency` | Валюта |
| `posted` | Проведён |
| `reserved` | Активен резерв (`РезервироватьТовар` и проведён) |
| `shipmentDate` | Дата отгрузки |
| `deliveryNumber`, `deliveryDate` | Расходная накладная (если создана) |
| `returnNumber`, `returnDate` | Возврат от покупателя по накладной (если создан) |
| `status`, `statusSetBy`, `statusDate` | Статус заказа |
| `blocked`, `canceled`, `readyToShip` | Признаки заказа (см. [Статусы заказа](#статусы-заказа)) |
| `items[]` | Позиции: `item`, `itemCode`, `qty`, `price` |

## PATCH /orders — полная замена заказа

Полностью заменяет заголовок (при наличии полей) и **весь** состав `items`. Требует массив `items` в теле.

Допустимые поля заголовка: `customer`/`customerCode`, `warehouse`/`warehouseCode`, `reserve`/`reserved`, `readyToShip`.

Алгоритм: снятие проведения → замена данных → повторное проведение (статус переходит в **«Изменен»**) → применение `readyToShip`, если передан. При ошибке проведения заказ восстанавливается в исходное состояние.

**Успешный ответ (200):** `success`, `orderNumber`, `incomingNumber`, `orderDate`, `reserved`, `posted`, `shipmentDate`, `status`, `statusDate`, `readyToShip`, `cancelled`.

---

## PUT /orders — добавление позиций

Добавляет позиции из `items` к существующему заказу. Остатки проверяются по **суммарному** количеству (старые + новые строки). Цены сверяются с прайсом. Проведение заказа переводит статус в **«Изменен»**.

Параметр `readyToShip` в **PUT /orders** не поддерживается — используйте **POST /status**.

**Успешный ответ (200):** `success`, `orderNumber`, `incomingNumber`, `orderDate`, `reserved`, `posted`, `shipmentDate`, `status`, `statusDate`, `readyToShip`, `cancelled`.

---

## DELETE /orders — удаление позиций

Тело: `items[]` с `itemCode` или `item` — номенклатура для удаления.

| Сценарий | Поведение |
|----------|-----------|
| Указаны **все** позиции заказа | Пометка документа на удаление, `deleted: true`, резерв снимается |
| Указана **часть** позиций | Удаление строк, перепроведение (статус → «Изменен»); пустой заказ **запрещён** → **400** |

**Успешный ответ (200):** `success`, `orderNumber`, `incomingNumber`, `orderDate`, `reserved`, `posted`, `deleted`, `shipmentDate`, `status`, `statusDate`, `readyToShip`, `cancelled`.

---

## Структура позиции (`items[]`)

Используется в **POST**, **PATCH**, **PUT**, **DELETE** `/orders`.

| Поле | Обяз. | Описание |
|------|-------|----------|
| `itemCode` | да* | Код номенклатуры |
| `item` | да* | Наименование номенклатуры |
| `qty` | нет | Количество (> 0, по умолчанию 1) |
| `price` | да | Цена продажи; должна совпадать с `salePrice` из **GET /balances** |

\* Достаточно одного из `itemCode` / `item`.

Поля `serial`, `batch`, `batchCode` в `/orders` **не поддерживаются** — серии указываются отдельно при отгрузке (**POST /delivery**).

Услуги не участвуют в проверке остатков.

## GET /series — серии для отгрузки

```
GET /series?orderNumber=MT-20260708-00042
GET /series?incomingNumber=PO-2026-001
```

Возвращает серийные номера на складе заказа только для позиций с серийным учётом:

```json
{
  "orderNumber": "MT-20260708-00042",
  "incomingNumber": "PO-2026-001",
  "items": [
    {
      "itemCode": "12345",
      "serials": ["SN001", "SN002"]
    }
  ]
}
```

## POST /delivery — создание расходной накладной

Создаёт и **проводит** расходную накладную на основании проформы. Применяется для заказов с **серийным** учётом.

| Поле | Обяз. | Описание |
|------|-------|----------|
| `orderNumber` / `incomingNumber` | да | Заказ-основание |
| `serials` | да | Массив серий для отгрузки (строки или числа) |

**Условия:**
- заказ проведён и является проформой (не финальным инвойсом);
- расходная накладная ещё **не создана**;
- все позиции заказа ведут серийный учёт;
- каждая серия из `serials` найдена на складе заказа и соответствует составу заказа;
- количество серий совпадает с количеством по позициям.

**Успешный ответ (200):**

```json
{
  "success": true,
  "number": "РН-000123"
}
```

---

## PATCH /delivery — блокировка (начало сборки)

Переводит заказ в статус **«Собирается (заблокирован)»**. Требует существующую расходную накладную по заказу — её создают в 1С до вызова API или через **POST /delivery**.

| Поле / параметр | Описание |
|-----------------|----------|
| `orderNumber` / `incomingNumber` | Заказ |
| `employeeCode` / `employeePin` | Оператор склада (рекомендуется) |

**Успешный ответ (200):**

```json
{
  "success": true,
  "orderNumber": "MT-20260708-00042",
  "incomingNumber": "PO-2026-001",
  "deliveryNumber": "РН-000123",
  "status": "Собирается (заблокирован)",
  "statusSetBy": "Петров П.П.",
  "statusDate": "2026-07-08T14:30:00"
}
```

Конфликт статуса → **409**.

---

## POST /status — установка признаков заказа

Устанавливает независимые булевы признаки заказа: **`blocked`**, **`canceled`**, **`readyToShip`**. Основной статус `status` при этом **не изменяется**. Расходная накладная не требуется (в отличие от **PATCH /delivery**).

| Поле | Обяз. | Описание |
|------|-------|----------|
| `orderNumber` / `incomingNumber` | да | Заказ (достаточно одного номера) |
| `blocked` | нет* | Признак блокировки для сборки (`true`/`false`, `1`/`0`, `yes`/`no`) |
| `canceled` | нет* | Признак отмены заказчиком |
| `readyToShip` | нет* | Признак готовности к отгрузке |
| `employeeCode` / `employeePin` | нет | Оператор склада |

\* Нужно указать **хотя бы один** из `blocked`, `canceled`, `readyToShip` — иначе **400**. Можно передавать несколько признаков одновременно.

`blocked`, `canceled`, `readyToShip` передаются **только в теле JSON**; `orderNumber`/`incomingNumber`/`employeeCode`/`employeePin` допускаются также в query string URL.

**Проверки:**
- заказ принадлежит учётной записи API;
- сотрудник (если указан) — оператор склада заказа;
- если **все** переданные признаки уже равны запрошенным значениям — ошибка (**409**, `КонфликтСтатуса`).

**Успешный ответ (200):**

```json
{
  "success": true,
  "orderNumber": "MT-20260708-00042",
  "incomingNumber": "PO-2026-001",
  "status": "Зарегистрирован",
  "statusSetBy": "Петров П.П.",
  "statusDate": "2026-07-08T14:30:00",
  "deliveryNumber": "",
  "returnNumber": "",
  "shipmentDate": "2026-07-10T00:00:00",
  "blocked": true,
  "canceled": false,
  "readyToShip": false
}
```

Если по заказу уже создана расходная накладная, поле `deliveryNumber` заполняется её номером.

| Код | Когда |
|-----|-------|
| **400** | Не указан ни один из `blocked`/`canceled`/`readyToShip`, не указан номер заказа, ошибка сотрудника, ошибка записи статуса |
| **404** | Заказ не найден или принадлежит другой учётной записи |
| **409** | Все переданные признаки уже установлены |

## GET /delivery-notes — печатная форма накладной

```
GET /delivery-notes?number=РН-000123&format=pdf
GET /delivery-notes?deliveryNumber=РН-000123&format=excel&serials=without
```

| Параметр | Синонимы | Описание |
|----------|----------|----------|
| `number` | `deliveryNumber`, `deliveryNoteNumber` | Номер расходной накладной |
| `date` | `deliveryNoteDate` | Дата накладной (опционально) |
| `format` | — | `pdf` (по умолчанию), `excel` / `xlsx` / `xls` |
| `serials` | — | `with` (по умолчанию) или `without`; также `true`/`false`, `1`/`0`, `yes`/`no` |

Ответ — бинарный файл (`application/pdf` или Excel) с заголовком `Content-Disposition: attachment`.

| Код | Когда |
|-----|-------|
| **200** | Успешная операция |
| **400** | Ошибка валидации, бизнес-правила (`success: false`, поле `error`) |
| **401** | Неверный или отсутствующий `X-Api-Key` |
| **404** | Заказ, накладная или сотрудник не найдены |
| **409** | Конфликт: заказ уже отгружен / отменён / заблокирован / есть накладная при редактировании / повторная установка статуса |
| **500** | Внутренняя ошибка 1С |

Типичный формат ошибки:

```json
{
  "success": false,
  "error": "Текст ошибки"
}
```

или `{ "error": "..." }` для части GET-эндпоинтов.

---

## Резервирование остатков

При `reserve: true` (значение по умолчанию) проведение проформы формирует движения резерва (регистр `ТоварыВДороге`, вид «Резерв»).

Поле `reserved` в ответах = `true`, если у документа включён `РезервироватьТовар` **и** документ проведён.

При создании и изменении заказа проверяется доступный остаток: `Остаток − Резерв` на складе (данные **GET /balances**).

## Примеры запросов

Ниже — сквозной сценарий интеграции. Базовый URL и ключ подставьте свои:

```
https://<host>/api/stock/...
X-Api-Key: your-api-token-here
```

Значения `itemCode`, `price`, `customerCode`, `warehouseCode`, `employeeCode` должны соответствовать данным вашей учётной записи. Цены берите из **GET /balances** (`salePrice`).

**Остатки и цены:**


**Склады учётной записи:**

```http
GET /warehouses
X-Api-Key: your-api-token-here
```

**Операторы склада:**

```http
GET /employees?warehouseCode=1
X-Api-Key: your-api-token-here
```

### 8. Отгрузка — POST /delivery

Создаёт и проводит расходную накладную. Все позиции заказа должны вести серийный учёт; в `serials` — полный перечень серий к отгрузке.

```http
POST /delivery
Content-Type: application/json
X-Api-Key: your-api-token-here
```

```json
{
  "incomingNumber": "PO-2026-001",
  "serials": ["SN001"]
}
```

**Ответ 200:**

```json
{
  "success": true,
  "number": "РН-000123"
}
```

### 9. Блокировка (начало сборки) — PATCH /delivery

Используется, когда расходная накладная **уже существует** (создана через **POST /delivery** или вручную в 1С), но сборка ещё не начата. Переводит заказ в статус «Собирается (заблокирован)».

```http
PATCH /delivery?employeeCode=42
Content-Type: application/json
X-Api-Key: your-api-token-here
```

```json
{
  "orderNumber": "MT-20260708-00042"
}
```

**Ответ 200:**

```json
{
  "success": true,
  "orderNumber": "MT-20260708-00042",
  "incomingNumber": "PO-2026-001",
  "deliveryNumber": "РН-000123",
  "status": "Собирается (заблокирован)",
  "statusSetBy": "Петров П.П.",
  "statusDate": "2026-07-08T14:30:00"
}
```

> **Примечание.** **POST /delivery** создаёт и **проводит** накладную (отгрузка) — после него **PATCH /delivery** вернёт **409**. **PATCH /delivery** используйте, когда накладная уже создана в 1С **без проведения** (например, оператором вручную), и нужно начать сборку.

### 10. Установка признаков заказа — POST /status

**Блокировка для сборки** (`blocked=true`):

```http
POST /status
Content-Type: application/json
X-Api-Key: your-api-token-here
```

```json
{
  "incomingNumber": "PO-2026-001",
  "blocked": true,
  "employeeCode": "42"
}
```

**Ответ 200:**

```json
{
  "success": true,
  "orderNumber": "MT-20260708-00042",
  "incomingNumber": "PO-2026-001",
  "status": "Зарегистрирован",
  "statusSetBy": "Петров П.П.",
  "statusDate": "2026-07-08T14:30:00",
  "deliveryNumber": "",
  "returnNumber": "",
  "shipmentDate": "2026-07-10T00:00:00",
  "blocked": true,
  "canceled": false,
  "readyToShip": false
}
```

**Снятие блокировки** (`blocked=false`):

```json
{
  "orderNumber": "MT-20260708-00042",
  "blocked": false
}
```

**Готовность к отгрузке** (`readyToShip=true`):

```http
POST /status?orderNumber=MT-20260708-00042&employeeCode=42
Content-Type: application/json
X-Api-Key: your-api-token-here
```

```json
{
  "readyToShip": true
}
```

**Отмена заказчиком** (`canceled=true`) — можно совмещать с другими признаками за один запрос:

```json
{
  "incomingNumber": "PO-2026-001",
  "canceled": true,
  "blocked": false,
  "employeeCode": "42"
}
```

**Ответ 200** (комбинированный запрос):

```json
{
  "success": true,
  "orderNumber": "MT-20260708-00042",
  "incomingNumber": "PO-2026-001",
  "status": "Зарегистрирован",
  "statusSetBy": "Петров П.П.",
  "statusDate": "2026-07-08T15:00:00",
  "deliveryNumber": "",
  "returnNumber": "",
  "shipmentDate": "2026-07-10T00:00:00",
  "blocked": false,
  "canceled": true,
  "readyToShip": false
}
```

**Не указан ни один признак — ответ 400:**

```json
{
  "success": false,
  "error": "Укажите хотя бы один параметр: blocked, canceled, readyToShip."
}
```

**Повторная установка уже установленных значений — ответ 409:**

```json
{
  "success": false,
  "error": "Запрошенные значения статуса уже установлены."
}
```
